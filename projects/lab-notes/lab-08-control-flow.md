## Exercise 5 — Benchmark Insight

The Result version was slower at both rates because in Python it allocates an `Ok` or `Err` object and does a `match` on every iteration, while the exception version’s success path is just a branch, a multiplication, and a plain return. The gap was larger at 0.1% invalid because almost every iteration took that very cheap non-exception path, so the per-iteration allocation and pattern-matching overhead of the Result version dominated even more clearly. In production Python, this means Result wrappers are valuable for clarity and explicit error handling but are not a zero-cost abstraction, so for very hot paths you should prefer lighter representations like `None`, sentinel values, or `(value, error)` tuples, and reserve exceptions for truly exceptional failures.
